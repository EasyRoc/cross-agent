# 商品分析 Agent — 技术方案文档

## 1. 项目概述

构建一个基于 **LangGraph Supervisor 模式** 的多 Agent 商品分析系统，支持从小红书、抖音、淘宝、亚马逊、得物等平台的商品数据进行多维度分析，最终生成结构化报告（Markdown/PDF）。

### 1.1 核心能力

| 能力 | 说明 |
|------|------|
| 意图识别 | 区分"商品分析"与"普通对话"，提取分析参数 |
| 多源数据采集 | RAG（Milvus 混合检索）、MCP 平台接口（含 Tavily 网络检索）、LLM 兜底生成 |
| 多维度分析 | 爆款/配色/价格带/品类/材质/风格/人群/购买动机/痛点/使用场景/购买路径/生命周期 |
| 报告生成 | Markdown / PDF，固定格式规范 |
| Human-in-the-Loop | 用户确认概览 → 迭代改进 → 生成终稿 |
| 流式输出 | Agent 推理过程与结果流式返回 |
| 记忆管理 | 短期对话历史 + 自动摘要记忆 |

## 3. 组件详细设计

### 3.1 Supervisor Agent（监督 Agent）

Supervisor Agent 是整个系统的编排核心，负责：

#### 3.1.1 意图识别

使用 LLM（阿里百炼）判断用户输入类别：

```
系统 Prompt:
你是一个商品分析系统的意图识别器。
判断用户输入属于以下哪一类：
1. product_analysis — 商品分析请求（如 "男式衬衫爆款分析"、"运动鞋2025趋势"）
2. normal_chat — 普通对话（如 "你好"、"今天天气怎么样"）

输出 JSON: {"intent": "product_analysis" | "normal_chat", "confidence": 0.0~1.0}
```

- **阈值策略**：confidence >= 0.7 时视为分析请求，否则视为普通对话直接回复。
- **参数提取**：识别为分析请求后，提取以下参数：

```json
{
  "product": "男式衬衫",
  "platforms": ["淘宝", "抖音"],
  "dimensions": ["爆款", "价格带", "配色", "材质"],
  "time_range": "近30天",
  "custom_time": null
}
```

时间范围默认逻辑：未申明时间 → 默认近30天。

#### 3.1.2 任务编排

- 使用 LangGraph 的 `StateGraph` 构建有向图
- 将分析任务分解为并行子任务
- 收集子 Agent 结果，汇总为分析概览
- 控制 Human-in-the-Loop 循环

#### 3.1.3 状态管理

```python
class AgentState(TypedDict):
    # 对话状态
    messages: list               # 短期记忆
    summary: str                 # 摘要记忆（超出窗口后启用）
    
    # 分析任务状态
    intent: str                  # 意图识别结果
    params: dict                 # 提取的参数
    status: str                  # pending / overview_generated / confirmed / rejected / completed / terminated
    
    # 数据
    raw_data: list               # 原始采集数据
    analysis_results: dict       # 各维度分析结果
    overview: str                # 分析概览（给用户确认用）
    final_report: str            # 最终报告
    report_format: str           # markdown / pdf
    
    # 多轮确认
    iteration: int               # 迭代次数
    feedback: str                # 用户反馈
```

### 3.2 四个子 Agent

#### Agent 1: DataCollector Agent（数据采集 Agent）

**职责**：从多渠道并行采集商品数据。

**数据源优先级**：
1. **RAG 检索**（高优先级）— 公司私有商品库
2. **MCP 平台接口** — 各电商平台实时数据 + Tavily 网络检索
3. **LLM 生成**（兜底）— 当以上来源均无数据时

**RAG 实现方案**：

```
用户查询 → Query Rewrite（LLM）
    → Dense Embedding（text2vec/bge）
    → Sparse Embedding（BM25/Splade）
    → Milvus Hybrid Search
        (dense_weight=0.7, sparse_weight=0.3)
    → Rerank（cross-encoder）
    → Top-K 结果
```

- **向量维度**：1024（dense），稀疏向量由 BM25 算法生成
- **混合检索权重**：dense=0.7，sparse=0.3（可调）
- **Embedding 模型**：阿里百炼 `text-embedding-v3` 或开源 `bge-m3`
- **Milvus Collection 设计**：

```json
{
  "collection_name": "product_catalog",
  "fields": [

    {"name": "product_name", "type": "varchar", "max_length": 256},
    {"name": "platform", "type": "varchar", "max_length": 64},
    {"name": "category", "type": "varchar", "max_length": 128},
    {"name": "price", "type": "float"},
    {"name": "sales_volume", "type": "int64"},
```

**Tavily MCP 集成说明**：

Tavily 作为网络搜索引擎 MCP，用于补充各电商平台无法覆盖的外部信息：

```python
# Tavily MCP 实现要点
# 1. 使用 Tavily Search API 进行网络检索
# 2. 返回结果映射为统一的 Product 数据模型
# 3. 优先级：仅当 RAG 或电商 MCP 数据不足时，或用户明确要求"全网"数据时启用

from tavily import TavilyClient

class TavilyMCP(BaseMCP):
    def __init__(self):
        self.client = TavilyClient(api_key=os.getenv("TAVILY_API_KEY"))
    
    async def search_products(self, query: str, time_range: str = "近30天", limit: int = 10) -> list[Product]:
        # 构建带时间过滤的搜索查询
        search_query = f"{query} 商品分析 市场趋势"
        
        response = self.client.search(
            query=search_query,
            search_depth="advanced",
            max_results=limit,
            include_domains=["xiaohongshu.com", "douyin.com", "taobao.com", ...],
            time_range=self._map_time_range(time_range)
        )
        
        # 将 Tavily 结果映射为 Product 模型
        return [self._to_product(result) for result in response.results]
    
    def _map_time_range(self, time_range: str) -> str | None:
        mapping = {"近7天": "week", "近30天": "month", "近90天": "quarter"}
        return mapping.get(time_range)
    
    def _to_product(self, result: dict) -> Product:
        return Product(
            platform="tavily",
            name=result.get("title", ""),
            description=result.get("content", ""),
            source_url=result.get("url", ""),
            ...
        )
```

#### Agent 2: MultiDimAnalyzer Agent（多维分析 Agent）

**职责**：对采集到的数据，从 12 个维度进行全面分析。

**分析维度矩阵**：

| 维度 | 分析内容 | 输出示例 |
|------|----------|----------|
| 爆款 | 销量排名、增长率、爆款特征 | TOP5 商品列表，爆款共性分析 |
| 配色 | 热门色系、色彩分布 | 主色调分布饼图，流行色趋势 |
| 价格带 | 价格区间分布、性价比分析 | 价格带柱状图，各区间销量占比 |
| 品类 | 子品类分布、品类关联 | 品类树，交叉销售分析 |
| 材质 | 材质分布、新兴材质 | 材质占比，新材料趋势 |
| 风格 | 风格分类、风格演变 | 主流风格TOP5，风格迁移趋势 |
| 人群 | 目标人群画像 | 年龄/性别/地域分布，消费力分层 |
| 购买动机 | 驱动因素、决策因素 | 功能性/情感性/社交性动机分析 |
| 痛点 | 用户不满、退货原因 | Top N 痛点列表，改进方向 |
| 使用场景 | 场景分类、场景频率 | 主要场景图谱，场景-品类关联 |
| 购买路径 | 触点分析、决策链路 | 搜索→浏览→比较→购买转化漏斗 |
| 生命周期 | 新品/成长期/成熟期/衰退期 | 各阶段商品占比，趋势判断 |

**实现方式**：

```python
# 每个维度通过独立的 LLM 调用进行分析
async def analyze_dimension(dimension: str, data: list[Product], params: dict) -> dict:
    prompt = f"""
    你是商品分析专家。请对以下商品数据进行 {dimension} 维度分析。
    
    分析要求：
    - 时间范围：{params['time_range']}
    - 平台：{params['platforms']}
    - 品类：{params['product']}
    
    商品数据：
    {json.dumps(data, ensure_ascii=False, indent=2)[:8000]}
    
    请输出结构化分析结果（JSON格式）。
    """
    return await call_llm(prompt)
```

**职责**：审核分析结果的质量、完整性和准确性，提出改进建议。

### 3.3 LangGraph Supervisor 实现

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, Literal

# 定义状态
class AgentState(TypedDict):
    messages: list
    summary: str
    intent: str
    params: dict
    status: str  # pending | overview_generated | confirmed | rejected | completed | terminated
    raw_data: list
    analysis_results: dict
    overview: str
    final_report: str
    report_format: str
    iteration: int
    feedback: str

# 构建图
workflow = StateGraph(AgentState)

# 添加节点
workflow.add_node("intent_recognition", intent_recognition_node)
workflow.add_node("data_collector", data_collector_node)
workflow.add_node("multi_dim_analyzer", multi_dim_analyzer_node)
workflow.add_node("quality_review", quality_review_node)
workflow.add_node("overview_generator", overview_generator_node)
workflow.add_node("final_report_generator", final_report_generator_node)
workflow.add_node("human_feedback", human_feedback_node)

# 添加入口边
workflow.set_conditional_entry_point(
    intent_recognition_node,
    {
        "product_analysis": "data_collector",
        "normal_chat": END
    }
)

# 数据采集后进入并行分析
workflow.add_edge("data_collector", "multi_dim_analyzer")

# 分支：质量审核
workflow.add_conditional_edges(
    "multi_dim_analyzer",
    quality_review_node,
    {
        "pass": "overview_generator",
        "need_recollect": "data_collector",
        "need_reanalyze": "multi_dim_analyzer"
    }
)

# 概览 → 用户确认
workflow.add_edge("overview_generator", "human_feedback")

# Human-in-the-Loop 循环
workflow.add_conditional_edges(
    "human_feedback",
    decide_next_step,
    {
        "confirmed": "final_report_generator",
        "rejected": "multi_dim_analyzer",   # 根据反馈重新分析
        "terminated": END
    }
)

workflow.add_edge("final_report_generator", END)

# 编译图
app = workflow.compile()
```

**并行执行实现**：

```python
# 使用 LangGraph 的 fan-out / fan-in 实现并行
from langgraph.lib import send

def multi_dim_analyzer_node(state: AgentState):
    """为每个启用的维度并行生成分析任务"""
    dimensions = state["params"].get("dimensions", ALL_DIMENSIONS)
    return [
        send("analyze_single_dim", {
            "dimension": dim,
            "data": state["raw_data"],
            "params": state["params"]
        })
        for dim in dimensions
    ]

# 所有维度分析完成后汇总
def aggregate_dimensions(state: AgentState):
    dimensions_results = {}
    for result in state["dimension_results"]:
        dimensions_results[result["dimension"]] = result
    state["analysis_results"] = dimensions_results
    return state
```

### 3.4 Human-in-the-Loop 实现

```
用户确认流程：

1. Supervisor 生成概览 → 流式返回给用户
2. 用户选择：
   a. ✅ 确认 → 进入最终报告生成
   b. ❌ 拒绝 + 输入改进建议 → 重新分析（iteration++）
   c. 🚫 终止 → 退出当前分析流程
3. 每次重新分析时，QualityReview Agent 参考用户反馈改进
4. 最大迭代次数：3 次（防止无限循环）
```

### 3.5 记忆系统

**短期记忆**：当前会话的最近 N 轮对话（N=20）。

**摘要记忆**（超出窗口时启用）：

```python
from langchain.memory import ConversationSummaryMemory

memory = ConversationSummaryMemory(
    llm=bailian_llm,
    max_token_limit=2000,
    memory_key="summary"
)

# 当消息数量 > 20 时，触发摘要
if len(state["messages"]) > 20:
    summary = memory.predict_new_summary(
        state["messages"][:-10],  # 除最近10轮外的历史
        state.get("summary", "")
    )
    state["summary"] = summary
    state["messages"] = state["messages"][-10:]  # 保留最近10轮
```

### 3.6 流式返回

使用阿里百炼的流式接口 + FastAPI WebSocket：

```python
from openai import AsyncOpenAI  # 阿里百炼兼容 OpenAI SDK

client = AsyncOpenAI(
    api_key=os.getenv("DASHSCOPE_API_KEY"),
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1"

async def stream_analysis(query: str, websocket: WebSocket):
    stream = await client.chat.completions.create(
        model="qwen-plus",  # 或 qwen-max
        messages=[...],
        stream=True
    )
    
    async for chunk in stream:
        if chunk.choices[0].delta.content:
            await websocket.send_json({
                "type": "token",
                "content": chunk.choices[0].delta.content
            })
    
    await websocket.send_json({"type": "done"})
```

### 3.7 LLM 配置（阿里百炼）

| 用途 | 模型 | 说明 |
|------|------|------|
| 意图识别 | `qwen-plus` | 快速、低成本 |
| 数据分析 | `qwen-max` | 推理能力强，适合复杂分析 |
| 报告生成 | `qwen-max` | 长文本生成 |
| Embedding | `text-embedding-v3` | 高维向量化 |
| 摘要记忆 | `qwen-plus` | 中等推理任务 |

### 3.8 Milvus 配置

```python
from pymilvus import connections, Collection

# 连接本地 Milvus（Docker）
connections.connect(
    alias="default",
    host="localhost",
    port="19530"
)

# 混合检索实现
async def hybrid_search(query: str, top_k: int = 20):
    # 1. 生成密集向量
    dense_emb = await embedding_model.embed_query(query)
    
    # 2. 生成稀疏向量（BM25）
    sparse_emb = bm25_model.encode(query)
    
    # 3. 混合检索
    collection = Collection("product_catalog")
    collection.load()
    
    results = collection.hybrid_search(
        reqs=[
            AnnSearchRequest(dense_emb, "dense_vector", params={"nprobe": 10}),
            AnnSearchRequest(sparse_emb, "sparse_vector", params={})
        ],
        rerank=RRFRanker(),
        limit=top_k
    )
    
    return results
```

## 6. 关键技术决策

### 6.1 为什么用 LangGraph 而非直接调 Agent？

| 特性 | LangGraph | 直接调 Agent |
|------|-----------|--------------|
| 状态管理 | 内置 StateGraph，状态持久化 | 需自行管理 |
| 循环控制 | 原生支持 cycle 和 conditional edge | 需手写循环 |
| 并行执行 | fan-out/fan-in 模式 | 需自行编排 |
| Human-in-the-Loop | 支持中断/恢复 | 需额外实现 |

### 6.2 为什么用 Milvus 混合检索？

- 密集向量捕捉语义相似性（如"男式衬衫"≈"商务衬衫"）
- 稀疏向量捕捉关键词精确匹配（如品牌名"海澜之家"）
- 结合两者可同时覆盖"语义理解"和"精确匹配"场景

### 6.3 为什么用阿里百炼？

- 国内合规：数据不出境
- 兼容 OpenAI SDK：迁移成本低
- 支持流式输出：适合实时交互场景
- 模型丰富：qwen-plus/qwen-max/text-embedding-v3 覆盖全场景

## 10. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| MCP 平台接口不稳定 | 数据采集失败 | 增加重试机制 + LLM 兜底 |
| LLM 分析结果幻觉 | 报告不准确 | QualityReview Agent 交叉验证 |
| 长会话 Token 超限 | 上下文丢失 | 摘要记忆 + 滑动窗口 |
| Milvus 检索延迟高 | 响应慢 | 增加缓存层 + 异步查询 |
| 用户无限循环拒绝 | 资源浪费 | 最大迭代次数限制（3次） |
